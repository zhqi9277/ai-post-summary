# ai-post-summary

`AI Post Summary` 是一个面向 WordPress 文章站点的 AI 内容增强插件。  
它不只生成文章摘要，还能补全文章标签、SEO 关键词，并提供后台管理、批量任务、失败重试、日志追踪和前端展示能力。

这个插件最初就是按真实运营站点需求做的，不是只演示一次 AI 调用，而是围绕“生成之后怎么展示、怎么批量补、怎么排错、怎么人工修正”整套流程来设计。

## 这个插件解决什么问题

很多 WordPress 站点在内容运营上会遇到几个典型问题：

- 文章很多，但摘要没有统一规范，文章页信息密度不够
- 标签经常漏填，后期归档和推荐效果差
- SEO 关键词字段经常空着，或者写得很随意
- 站点里既有人工发文，也有采集或自动发布流程，人工逐篇补内容成本太高
- AI 调用不是最难的，难的是生成失败以后怎么查、怎么补、怎么重试

`ai-post-summary` 的目标不是“让 AI 调一下接口”，而是把这条链路做成一个能长期用的后台能力。

## 核心优势

### 1. 不只生成摘要，而是生成可运营的文章附加信息

插件现在支持三类内容：

- AI 摘要
- WordPress 文章标签 `post_tag`
- 文章 SEO 关键词 `keywords`

这意味着它不是一个单点摘要插件，而是一个针对文章页的 AI 增强工具。

### 2. 兼顾自动化和人工可控

很多 AI 插件要么完全自动，要么完全手动。这个插件两边都做了：

- 可以在文章发布或更新后自动排队生成
- 可以在文章编辑页手动立即生成或重新生成
- 可以在管理页里手工修改摘要、标签、关键词
- 可以批量补齐历史文章
- 可以重试失败任务

这套设计更适合真实运营，而不是只适合演示。

### 3. 有管理后台，不是“生成完就结束”

插件带完整后台管理能力：

- 设置页
- 摘要管理
- 标签 / 关键词管理
- 批量任务
- 日志页

你可以查看、编辑、重试、筛选，不需要每次都进文章编辑页逐篇处理。

### 4. 前台展示和站点风格融合，而不是独立漂浮组件

摘要模块支持：

- 小工具插入
- 短代码插入
- 正文顶部自动插入

同时已经针对 `zibll` 主题做过兼容处理，前端卡片样式更接近主题自身模块，而不是一块突兀的默认插件面板。

### 5. 有日志，方便排错

这类功能如果没有日志，出了问题就很难定位。插件支持：

- `DEBUG`
- `INFO`
- `WARNING`
- `ERROR`

并且日志会持久化保存，方便追踪：

- 自动排队
- 手动生成
- 批量补齐
- 接口调用失败
- 返回为空
- 解析失败
- 生成成功

### 6. 不是只适配一种 AI 厂商

插件使用兼容 `Chat Completions` 请求格式的接口，因此可以接入多种兼容服务，例如：

- OpenAI 兼容接口
- DeepSeek 兼容接口
- NVIDIA Integrate
- 其他支持 Chat Completions 的服务

只要接口、模型名和 API Key 正确即可。

## 与普通“AI 摘要插件”的差异

很多现成插件只做一件事：在前台加载文章时临时调一次 AI，然后把结果显示出来。这样的问题很明显：

- 首屏变慢
- 重复消耗 token
- 请求失败直接影响页面
- 没有批量补齐能力
- 没有后台管理能力

`ai-post-summary` 采用的是另一种思路：

- 生成走后台或异步任务
- 前台只读取已生成结果
- 失败不拖垮文章页
- 后台可以重试和编辑
- 支持批量补历史内容

这更像一个“站点内容工作流工具”，而不是一个纯前台展示插件。

## 适合哪些站点

- 内容资讯站
- 博客站
- 采集站
- 运营型 SEO 站
- 需要统一文章信息结构的站点
- 使用 `zibll` 主题的站点

如果站点有大量历史文章、标签补全需求、关键词补全需求，这个插件会比只做摘要的工具更有价值。

## 安装方式

### 方式一：直接上传插件目录

1. 将 `ai-post-summary` 目录上传到 `/wp-content/plugins/`
2. 在 WordPress 后台启用插件
3. 进入 `设置 -> AI 摘要`
4. 填写接口地址、API Key、模型名

### 方式二：从仓库拉取

```bash
git clone https://github.com/zhqi9277/ai-post-summary.git
```

然后将插件目录放入：

```bash
/wp-content/plugins/ai-post-summary
```

### 接口地址说明

插件会直接向你填写的地址发起请求，因此要填写完整接口地址，而不是只填根路径。

例如：

```text
https://api.openai.com/v1/chat/completions
```

如果接的是 NVIDIA Integrate，也要填完整接口，而不是只填 `/v1/`。

## 前台使用方式

### 1. 小工具

适合主题本身有文章页小工具位的站点。

在后台：

- `外观 -> 小工具`

把：

- `AI 文章摘要`

拖到文章页对应位置即可。

### 2. 短代码

可以手动插入：

```shortcode
[ai_post_summary]
```

或者：

```shortcode
[ai_post_summary post_id="123"]
```

### 3. 正文自动插入

开启后会自动插到文章正文顶部。

## 数据写入方式

### 摘要

摘要会写入文章 meta，例如：

- `_ai_summary`
- `_ai_summary_status`
- `_ai_summary_source_hash`
- `_ai_summary_updated_at`
- `_ai_summary_error`

### 标签

标签写入 WordPress 原生文章标签：

- `post_tag`

### 关键词

关键词写入文章 meta：

- `keywords`

这个字段和 `zibll` 主题的 SEO 关键词逻辑兼容。

## 生成逻辑说明

### 摘要

- 提取文章正文纯文本
- 去掉脚本、样式、短代码
- 限制文本长度
- 调用 AI
- 对结果做格式清洗
- 根据设置的上限截断
- 写回文章 meta

### 标签 / 关键词

- 读取文章标题和正文
- 请求 AI 返回 JSON
- 解析 `keywords` 和 `tags`
- 清洗重复和空值
- 控制数量上限
- 写回关键词 meta 和 WordPress 标签

## 为什么要支持自定义 Prompt

不同站点的摘要风格和 SEO 策略差异很大：

- 资讯站更适合导读式摘要
- 技术站更适合要点式摘要
- SEO 站更关注关键词覆盖
- 社区站更关注标签归类

所以插件允许在设置页中配置：

- 系统提示词
- 摘要模板

这样你可以根据站点定位调整输出风格，而不需要每次改代码。

## 日志为什么重要

这类插件上线之后，真正耗时间的不是“第一次生成成功”，而是后续维护：

- 为什么某篇文章没生成
- 为什么某个模型返回空内容
- 为什么某篇文章一直失败
- 批量任务到底排了多少篇

没有日志，这些问题很难排。  
因此插件把日志做成了后台可查看、可筛选、可清空的持久化能力。
